# Plan — Reusable Reaction Editor (web Inspector)

**Status:** ready to implement (designed 2026-06-26). The engine needs **no
changes**. This is a web + thin API-metadata feature.

## Problem
The `reaction` / `reactions` parameters on reactor unit ops are structured
(JSON) values. The web guided param editor (`Inspector.tsx → SchemaParamRow`)
renders any `complex`/`json` param **read-only** ("— set via .flow / Copilot"),
so a user **cannot build a reactor flowsheet from the palette** — the ammonia
loop (and every reactor template) can only come from a `.flow` or Copilot.
Discovered while driving Tutorial 2 (ammonia loop).

## Goal
**One** reusable `ReactionEditor` modal, driven by a per-param **capability
descriptor**, that lets the user define reactions for **all** reaction-bearing
units from the UI — not a bespoke editor per unit.

## Scope (the 4 units that take user-defined reactions)
| Unit | Param(s) today | Reaction kind | Notes |
|------|----------------|---------------|-------|
| `EquilibriumReactor` | `reaction` or `reactions` (first used) | **stoichiometric** `{stoich, key}` | single reaction; extent from K_eq (no conversion field) |
| `ConversionReactor`  | `reaction`+`conversion`, or `reactions` (each `{stoich,key,conversion}`) | **stoichiometric** + per-reaction `conversion` | multiple reactions allowed, applied in order |
| `CSTR` | `reactions` (required) | **kinetic** `KineticReaction` | multiple; needs `V`, optional `T` |
| `PFR`  | `reactions` (required) | **kinetic** `KineticReaction` | multiple; needs `V`, `n_steps`, optional `T` |

**Out of scope:** `GibbsReactor` (Cantera Gibbs-min, no stoichiometry) and
`ClausReactor` (reactions hardcoded). Neither exposes a reaction param.

**Future-free:** any later unit that exposes a `reactions` param (e.g. a
reactive `RigorousColumn`, which already uses `KineticReaction` internally per
`reaction.py`) is covered automatically once its schema entry is tagged.

## Canonical data shapes (source of truth: `engine/caldyr/unitops/reaction.py`)
```
Reaction.from_param:         {"stoich": {comp: signed_coeff}, "key": reactant_id?}
                             # ConversionReactor list form adds: "conversion": 0..1
KineticReaction.from_param:  {"stoich": {...}, "key": reactant_id,    # key REQUIRED
                              "k0": float, "Ea": float_J_per_mol,
                              "orders": {comp: order}?,               # default {key: 1}
                              "k0_rev": float?, "Ea_rev": float?,     # reversible (optional)
                              "orders_rev": {comp: order}?}           # default = product stoich
```
- Signed coefficients: **reactants negative, products positive**.
- `key` must be a reactant (negative coeff). Required for kinetic & for
  conversion; optional (but recommended) for equilibrium.

### Unifying decision — always WRITE the list form `reactions`
The engine accepts `reactions` (list) for **all four** units, so the editor
**always serializes to `reactions`** and clears the singular `reaction` /
top-level `conversion` keys on save. On **open**, normalize singular→list:
- `reaction` (+ unit-level `conversion`) → `[{...stoich, key, conversion?}]`.
This gives one read/write path. (EquilibriumReactor uses only the first entry;
the editor restricts it to a single reaction — see `multiple` flag.)

## Design — capability descriptor (server-tagged, web-thin)
Add an `editor` discriminator + `editor_opts` to the reaction param entries in
**`api/param_schemas.py`** (the API stays the single source of truth; the web
just renders whatever descriptor it receives). Restart the API after editing.

Tag these entries:
```python
# EquilibriumReactor.reaction AND .reactions:
'editor': 'reaction',
'editor_opts': {'kind': 'stoichiometric', 'multiple': False,
                'conversion': False, 'key_required': False},
# ConversionReactor.reaction AND .reactions:
'editor': 'reaction',
'editor_opts': {'kind': 'stoichiometric', 'multiple': True,
                'conversion': True, 'key_required': True},
# CSTR.reactions AND PFR.reactions:
'editor': 'reaction',
'editor_opts': {'kind': 'kinetic', 'multiple': True,
                'conversion': False, 'key_required': True, 'reversible': True},
```
`editor_opts` fields: `kind` ('stoichiometric'|'kinetic'), `multiple` (bool — Add
reaction allowed), `conversion` (bool — per-reaction conversion field),
`key_required` (bool), `reversible` (bool — kinetic reverse params, kinetic only).

> Note: param_schemas.py says it is GENERATED by `build_param_schemas.py`, but
> that script is **not in the repo** — the file is hand-maintained in practice.
> Edit it directly; if the generator is ever restored, port these tags into it.

## UI (chosen: two-column reactants | products, live preview)
New `web/src/components/ReactionEditorDialog.tsx` — a modal in the
ScenariosDialog/GlossaryDialog style (store-driven open state). Layout:

- **Header:** `Edit reactions — <unitType>`; Save / Cancel; X/Esc to close.
- **Per reaction (card):**
  - **Reactants** column + **Products** column. Each row = {component `<select>`
    from `store.components`} × {positive magnitude `NumberInput`}; +add / remove
    per row. Editor applies signs (reactants → negative).
  - **Key reactant:** radio/`<select>` limited to the reactant species. Shown
    when `key_required` or `kind==='kinetic'`; optional otherwise.
  - **conversion** (0..1) `NumberInput` — only if `editor_opts.conversion`.
  - **kinetic block** (only if `kind==='kinetic'`): `k0`, `Ea` (J/mol), and an
    **orders** sub-list {component, order} (defaults to `{key:1}`); a
    **reversible** toggle revealing `k0_rev`, `Ea_rev`, `orders_rev`.
  - **Live preview line:** `N₂ + 3 H₂ → 2 NH₃` (built from current rows; uses
    Unicode subscripts if easy, else plain).
  - Remove-reaction button.
- **Add reaction** button — only if `editor_opts.multiple`.
- **Validation (block Save, inline messages):** each reaction has ≥1 reactant &
  ≥1 product; no component on both sides unless intended; key is a listed
  reactant; conversion ∈ (0,1]; kinetic `k0>0`, `Ea≥0`; orders reference real
  components.
- **Save →** `setParam(nodeId, 'reactions', list)` then
  `unsetParam(nodeId, 'reaction')` and `unsetParam(nodeId, 'conversion')`.

  entries (EquilibriumReactor ×2, ConversionReactor ×2, CSTR ×1, PFR ×1).
  Restart API (`python -m uvicorn api.main:app --port 8753`).

## Acceptance / validation
1. **From an empty canvas, build the ammonia reactor in-UI:** components N₂/H₂/
   NH₃/Ar, PR; add EquilibriumReactor; open the reaction editor; reactants N₂×1,
   H₂×3; product NH₃×2; key N₂; set `T=673.15`. Wire a feed; Solve → reactor
   outlet matches the template path.
2. **ConversionReactor** with a single reaction + `conversion=0.8` solves; a
   **two-reaction** ConversionReactor solves (extents applied in order).
3. **CSTR** with one kinetic reaction (`k0`, `Ea`, `orders`, `V`) solves; a
   **reversible** kinetic reaction round-trips and solves.
4. **Existing templates** (ammonia loop, VCM, DME — singular/`reactions` forms)
   still load, display correctly in the editor, and solve unchanged.
5. The full Tutorial-2 ammonia loop, now buildable end-to-end from the palette,
   reproduces: converged, tear=RECYCLE, PRODUCT 45.06 mol/s @ 98.2% NH₃, LCOP
   $0.69/kg.

## Deferred / stretch (note, don't silently drop)
- Live **atom-balance** hint in the editor (needs per-component element counts;
  would require the API to expose formulae — e.g. extend `/components`). Engine
  validates balance-agnostic; this is a friendliness add-on.
- Reuse the same store-driven modal pattern for other complex params
  (`ComponentSplitter.splits`, `Cyclone.solids`) — separate editors, later.
- Tag a reactive `RigorousColumn`'s `reactions` param if/when it exposes one.
